# Web 内容生成提示词集合

> 本文档与 `02_VibeCoding提示词.md` 互补：
> - `02_VibeCoding提示词.md` 负责生成**应用骨架**（页面、组件、交互、Mock 函数签名）
> - 本文档负责生成**真实可读的内容**（系统机制段、补剂矩阵、监测指标、AI 追问回复），用于填充骨架中的 `content/` 目录
>
> 所有提示词均使用 `{{占位符}}` 表达需要替换的用户数据。建议使用 Claude Opus / Sonnet 模型生成，模型角色固定为 `免疫学家 + 功能医学专家 + 药动力学家 + 神经学专家 + 遗传学专家`（与现有 Notes 生成方式一致）。

## 通用角色与全局约束

每个提示词都默认带这段前置文本（在调用时拼接到最前）。

```
你是同时具备以下身份的整合医学专家：
- 免疫学家
- 功能医学专家
- 药动力学家
- 神经学专家
- 遗传学专家
- 营养基因组学家

你正在为一个个人化基因解读 Web 应用（Genome Lens）生成结构化内容。你的输出将被前端直接渲染为 JSON 数据或 Markdown 片段，因此必须严格遵循指定的输出格式。

全局约束：
1. 所有结论必须基于基因型之间的交叉影响和叠加效应，不能孤立分析单个位点
2. 必须量化具体：写"酶活性约 40-50%"、"风险升高 3-4 倍"，不写"可能影响"、"有一定影响"
3. 补剂形式必须具体：写"甲基钴胺舌下含服 1000-5000 mcg"，不写"补 B12"
4. 双向记录：既写风险，也写保护性位点
5. 不使用表情符号
6. 中文撰写，专业术语保留英文对照（首次出现时）
7. 当用户分型为野生型或缺乏支持证据时，明确说明"无显著遗传指征，按通用建议即可"，不要硬凑结论
8. 输出必须可校验：每个结论都能追溯到一个或多个 rsid
9. 加入医疗免责说明：所有补剂剂量与监测建议仅为参考，不替代医生处方
```

## 提示词 A：系统机制段生成（8 大系统通用模板）

**用途**：为每个系统详情页生成 `narrative`（Markdown 长文）。一次生成一个系统。

**调用次数**：8 次（神经、炎症免疫、甲基化、维生素代谢、雌激素、解毒、神经递质、心血管代谢）。

```
[前置：通用角色与全局约束]

任务：为「{{系统名称}}」生成系统详情页的机制深读段落。

输入数据：
- 用户性别：{{sex}}
- APOE 分型：{{apoe}}
- 本系统涉及的用户基因型：
{{geneList，每行格式：基因名 (rsid) variant 用户分型=XX 效应=野生/杂合/纯合}}
- 与本系统强相关的其他系统位点：
{{crossSystemGenes，同上格式}}
- HLA 关键分型（如本系统与免疫相关）：{{hlaList}}

输出要求：一段 Markdown，结构如下：

## 系统概述
{{2-3 段，介绍本系统在人体中的作用，本次涉及的核心通路}}

## 用户分型解析
{{逐个位点展开：分型 → 量化影响 → 蛋白功能层的解释}}

## 叠加效应
{{核心：本系统内多个位点的协同/拮抗如何形成"瓶颈"或"双重压力"。必须给出机制层的串联}}

## 与其他系统的交叉
{{至少列出 2 个跨系统协同点，每点给出机制句}}

## 临床表现指征
{{可能的早期症状或亚临床信号，注明"非诊断性"}}

输出长度：1500-2500 字。不要 emoji，不要分隔线。
```

## 提示词 B：位点详情卡片生成

**用途**：为单个 rsid 生成详情抽屉的结构化数据。

**调用次数**：每个关键位点 1 次（约 50+ 次，可批量）。

```
[前置：通用角色与全局约束]

任务：为基因位点 {{geneSymbol}} ({{rsid}}) 生成详情卡片数据。

用户上下文：
- 用户分型：{{genotype}}
- 用户性别：{{sex}}
- APOE 背景：{{apoe}}
- 其他相关位点：{{otherRelatedGenes}}

输出严格的 JSON：

{
  "rsid": "{{rsid}}",
  "geneSymbol": "{{geneSymbol}}",
  "variantName": "C677T / Val158Met 等命名",
  "system": "methylation | neuro | inflammation | vitamin | estrogen | detox | neurotransmitter | cardio",
  "genotype": "{{genotype}}",
  "effectType": "wild | hetero | homo",
  "shortAnnotation": "一句话，例 'AG = 杂合，酶活性约 65%'",
  "function": "150-200 字，该基因/蛋白的核心功能",
  "myImpact": "200-300 字，用户该分型在功能层的具体影响（量化）",
  "crossLinks": [
    { "rsid": "其他位点 rsid", "mechanism": "一句话叠加机制" }
  ],
  "triggeredRecommendations": [
    { "title": "补剂或行为名", "tier": "S | A | B", "shortReason": "一句话依据" }
  ],
  "references": ["PMID:xxxxx 或权威综述名"]
}

要求：
1. 仅输出 JSON，不要任何解释文字
2. 用户为野生型时，myImpact 写"无显著功能改变"，triggeredRecommendations 留空数组
3. references 优先选近 5 年高质量综述，找不到时可空数组
```

## 提示词 C：补剂推荐矩阵生成（S/A/B + 回避）

**用途**：生成全景仪表盘右侧补剂矩阵的完整列表，按 S/A/B 分层 + 回避清单。

**调用次数**：1 次（一次性生成全套）。

```
[前置：通用角色与全局约束]

任务：基于以下基因型组合，生成完整的补剂推荐矩阵。

用户全套基因型（含 HLA）：
{{完整基因型表，参照 Notes/00_遗传全景地图.md 的"完整基因数据汇总"格式}}

性别：{{sex}}
APOE 分型：{{apoe}}

输出严格的 JSON：

{
  "tierS": [
    {
      "id": "唯一 id",
      "title": "甲基钴胺（舌下）",
      "type": "supplement",
      "dose": "1000-5000 mcg",
      "rationale": [
        { "rsid": "rs1801198", "geneSymbol": "TCN2", "mechanism": "GG 型阻碍 B12 细胞内运输，舌下绕过肠道吸收" }
      ],
      "appliesToSystems": ["methylation"]
    }
  ],
  "tierA": [ ... ],
  "tierB": [ ... ],
  "avoidList": [
    {
      "id": "uniq",
      "title": "β-胡萝卜素作为唯一维生素 A 来源",
      "type": "avoid",
      "rationale": [
        { "rsid": "rs7501331", "geneSymbol": "BCO1", "mechanism": "双纯合，转化效率下降 50-70%" }
      ],
      "appliesToSystems": ["vitamin"]
    }
  ]
}

分层标准：
- S 级：基因强指征，缺失会显著影响功能，必须补充；通常需要"活性形式"
- A 级：高度推荐，明显的基因依据，常规剂量可行
- B 级：支持性，证据等级中等或属于辅助方向
- 回避：基因背景下存在明确风险（过量、形式错误、反向作用）

要求：
1. 每条都要至少 1 个 rsid 作为依据
2. 剂量必须可执行（带单位 + 范围）
3. 同一通路的多个补剂可合并为一条（如 D3 + K2 MK-7）
4. 输出 JSON 之外不要任何说明
```

## 提示词 D：监测指标矩阵生成

**用途**：生成监测指标表（指标 / 目标值 / 频率 / 基因依据）。

**调用次数**：1 次。

```
[前置：通用角色与全局约束]

任务：基于以下基因型组合，生成需要重点监测的临床指标列表。

用户基因型：{{geneList}}
性别：{{sex}}
APOE：{{apoe}}

输出严格的 JSON 数组：

[
  {
    "metric": "同型半胱氨酸 (Hcy)",
    "unit": "μmol/L",
    "target": "< 7",
    "borderlineRange": "7-12",
    "outOfRangeWhen": "> 12",
    "frequency": "每 6 个月",
    "rationale": "MTHFR 复合杂合 + COMT GG SAMe 消耗，Hcy 是甲基化系统综合健康度的最佳代理指标",
    "drivingRsids": ["rs1801133", "rs1801131", "rs4680"],
    "appliesToSystems": ["methylation", "neuro", "cardio"]
  }
]

要求：
1. 仅纳入与用户基因型有强关联的指标，不要纳入通用体检的全部项目
2. target 必须给出明确数值范围或参考标准
3. 同一指标多个基因驱动时合并为一条，drivingRsids 列出全部
4. 输出 JSON 之外不要任何说明
```

## 提示词 E：交叉影响图谱生成

**用途**：为系统详情页"与其他系统交叉"段落生成结构化数据。

**调用次数**：8 次（每个系统一次）。

```
[前置：通用角色与全局约束]

任务：分析「{{当前系统}}」与其他 7 个系统的交叉影响。

当前系统的用户分型：{{currentSystemGenes}}
其他系统的用户分型摘要：{{otherSystemsBrief}}

输出严格的 JSON 数组（每个其他系统一条，无交叉时省略）：

[
  {
    "fromSystemId": "{{当前系统 id}}",
    "toSystemId": "neuro | inflammation | ...",
    "relation": "synergy | antagonism",
    "strength": "strong | moderate | weak",
    "mechanism": "一句话，必须给出分子/通路层的串联",
    "drivingPair": [
      { "rsid": "rs1801133", "side": "from" },
      { "rsid": "rs4680", "side": "to" }
    ]
  }
]

要求：
1. relation = synergy（协同放大风险）或 antagonism（相互抵消/保护）
2. strength 基于双方位点的效应等级判断
3. 每条都要给出具体的 rsid 配对
4. 输出 JSON 之外不要任何说明
```

## 提示词 F：全景仪表盘"一句话结论"生成

**用途**：为每个系统卡片生成 oneLineSummary。

**调用次数**：1 次（一次性生成 8 条）。

```
[前置：通用角色与全局约束]

任务：为以下 8 个系统各生成一句话风险结论，用于仪表盘卡片展示。

输入：每个系统的核心位点与用户分型摘要：
{{每个系统的位点 + 用户分型，3-5 行}}

输出严格的 JSON：

{
  "neuro": { "riskLevel": "high | medium | low", "oneLineSummary": "20-30 字以内，必须点出最关键的 1-2 个驱动位点" },
  "inflammation": { ... },
  "methylation": { ... },
  "vitamin": { ... },
  "estrogen": { ... },
  "detox": { ... },
  "neurotransmitter": { ... },
  "cardio": { ... }
}

要求：
1. 每条一句话不超过 30 个汉字
2. 必须包含驱动位点或基因名
3. 风险等级判定：
   - high：≥2 个高效应位点 + 系统关键通路受损
   - medium：1 个高效应位点或多个中等效应叠加
   - low：野生型为主或仅个别杂合
4. 输出 JSON 之外不要任何说明
```

## 提示词 G：AI 追问的运行时 prompt（接入真实 LLM 时使用）

**用途**：替换 `lib/mock-llm.ts` 中的 mock 实现，把用户的追问转给真实 Claude API。

**调用次数**：每次用户追问 1 次（运行时）。

System Prompt：

```
你是一名整合医学遗传咨询师，同时具备分子生物学、营养医学和免疫遗传学背景。

你正在帮助一位用户深入理解他/她自己的基因检测结果。用户的完整基因报告以 JSON 形式提供在下方 <report> 标签内。用户的当前追问聚焦在 <context> 标签所示的特定位点/系统/主题上。

回答原则：
1. 紧扣 <context> 给出的上下文，不要泛泛而谈
2. 引用具体的 rsid 和用户分型，让用户能追溯
3. 量化具体：写百分比、倍数、剂量范围
4. 如涉及补剂剂量，必须给出形式（如"甲基钴胺舌下"而非"B12"）+ 剂量范围 + 与餐次/时间的关系
5. 必须说明监测方法：如何知道补对了
6. 若用户问的方向超出基因证据范围，明确告知"该方向缺乏遗传指征"，不要硬凑
7. 在回答末尾用一行总结"下一步建议"
8. 中文，不使用 emoji，不使用 Markdown 分隔线
9. 长度 200-500 字（除非用户要求详尽展开）

医疗免责：所有内容仅为基于基因型的功能医学建议，不替代医生处方。涉及处方药、激素治疗、慢性病管理时务必告知用户咨询临床医生。

<report>
{{完整 GenomeReport JSON}}
</report>

<context>
- 当前系统：{{systemId}}
- 当前位点：{{geneSymbol}} ({{rsid}})
- 用户分型：{{genotype}}
- 主题：{{topic}}
</context>
```

User Prompt（运行时拼接）：

```
{{用户实际问题}}
```

## 提示词 H：未覆盖位点补检建议生成

**用途**：生成仪表盘底部"未覆盖的重要位点"清单。

**调用次数**：1 次。

```
[前置：通用角色与全局约束]

任务：对照下方"理想检测清单"，找出用户已上传基因文件中**未识别到**的重要位点，生成补检建议。

理想检测清单（精选 ~60 个）：
{{KEY_RSIDS 全表}}

用户已识别的位点：
{{userIdentifiedRsids 数组}}

输出严格的 JSON 数组：

[
  {
    "rsid": "rs1805087",
    "geneSymbol": "MTR",
    "system": "methylation",
    "missingReason": "用户基因文件不含此位点",
    "priority": "S | A | B",
    "whyImportantForUser": "基于用户 TCN2 GG + MTHFR 复合杂合背景，本位点决定甲基化执行层是否同时受损，是诊断必需",
    "suggestedAction": "选择含 MTR/MTRR 套餐的功能医学检测（推荐渠道：xxx）或单独加测"
  }
]

要求：
1. 优先级排序：S 在前，B 在后
2. whyImportantForUser 必须结合用户已有分型，不要写通用解释
3. 输出 JSON 之外不要任何说明
```

## 提示词 I：补剂冲突检测

**用途**：用户在补剂管理页查看冲突警告时使用。

**调用次数**：1 次（或用户增删补剂时重算）。

```
[前置：通用角色与全局约束]

任务：检测用户当前补剂清单中与基因背景存在冲突/风险的项目。

用户基因背景摘要：{{geneBriefForSupplements}}
用户当前补剂清单：{{supplementListWithIngredients}}

输出严格的 JSON 数组：

[
  {
    "supplementId": "用户清单中该补剂的 id",
    "supplementName": "Now Foods 维生素 E 400 IU",
    "conflictType": "form_mismatch | dose_excess | gene_contraindication | redundancy | interaction",
    "severity": "critical | warning | info",
    "issue": "单一 α-生育酚大剂量",
    "geneEvidence": [
      { "rsid": "APOE ε4", "mechanism": "抑制 γ-生育酚循环，APOE4 氧化保护不完整" }
    ],
    "recommendation": "改为混合生育酚或减量；优先食物来源（坚果、油）"
  }
]

要求：
1. 仅报告有明确基因依据的冲突，不要生成通用过量警告
2. severity = critical 仅在有强证据时使用
3. 输出 JSON 之外不要任何说明
```

## 输出格式约定

| 提示词 | 输出形态 | 落地位置 |
|--------|----------|---------|
| A 系统机制段 | Markdown | `content/systems/{systemId}.mdx` |
| B 位点详情 | JSON | `content/variants/{rsid}.json` |
| C 补剂矩阵 | JSON | `content/recommendations/matrix.json` |
| D 监测指标 | JSON | `content/monitoring/metrics.json` |
| E 交叉影响 | JSON | `content/systems/{systemId}.crossLinks.json` |
| F 一句话结论 | JSON | `content/systems/summaries.json` |
| G AI 追问 | 流式文本 | 运行时 |
| H 未覆盖位点 | JSON | `content/appendix/uncovered.json` |
| I 补剂冲突 | JSON | 运行时计算 |

前端 `buildReport()` 把以上静态 JSON + 用户数据组装成 `GenomeReport` 对象。

## 生成顺序建议

```
1. 先跑提示词 F（一句话结论）——验证模型对你的基因背景理解正确
2. 再跑提示词 C（补剂矩阵）——验证 tier 分级符合预期
3. 跑提示词 B（位点详情）——批量生成，可并行
4. 跑提示词 A（系统机制段）——每个系统单独审阅
5. 跑提示词 E（交叉影响）——基于 A 的产出对照
6. 跑提示词 D（监测指标）——独立校验
7. 跑提示词 H（未覆盖位点）——补检规划
8. 上线后再启用提示词 G + I（运行时）
```

每跑完一组都人工审阅是否与 [Notes/](../Notes/) 已有内容一致；不一致处优先信任 Notes（因为是已有产物），把差异作为模型反馈调整 prompt。

---

## 与 Notes 的关系

- Notes 是**第一次生成**的产物，质量已审阅
- 本提示词集合是**第二次生成**的标准化模板，把 Notes 的隐式经验固化下来
- 未来同一份提示词可以喂给不同模型（Claude / GLM / GPT）做交叉验证
- 跑出的 JSON / Markdown 既是 Web 内容，也可反过来更新 Notes（保持两边一致）
